## Introducción

Esta sección presenta la vista de datos del diseño según IEEE 1016-2009, transcrita de los esquemas y procedimientos almacenados de cada `init.sql`. Documenta la estrategia de persistencia de una base de datos MySQL por servicio con estado, la estrategia de multitenancy de esquema compartido con la columna discriminadora `TenantId` (con su justificación y la mitigación de la colisión parámetro/columna), el modelo entidad-relación detallado por servicio —cada esquema con su justificación de diseño, su representación en notación **DBML** (legible en [dbdiagram.io](https://dbdiagram.io)) y la explicación de las relaciones entre tablas— y un diccionario de datos trabajado sobre las tablas centrales de Auth. Cada campo se acompaña de su `ruta:línea` para que el lector pueda verificar la transcripción contra la fuente de verdad.

## 3.3.4.1 Estrategia de persistencia: una base de datos por servicio

Coherente con la arquitectura de microservicios (3.3.1), **cada backend con estado posee su propia
base de datos MySQL** y nadie accede a la base de otro servicio: la integración ocurre por gRPC y
eventos, nunca por tablas compartidas. Hay cuatro bases de datos (Auth, CourseManagement,
Attendance, Payment); Credentials no tiene base de datos por ser un servicio sin estado. No existen
claves foráneas que crucen servicios: cuando Attendance referencia una clase o un estudiante, lo
hace por su **identificador** (validado por gRPC o recibido por evento), no por una relación
referencial entre esquemas.

## 3.3.4.2 Estrategia de multitenancy: esquema compartido con `TenantId`

La decisión de diseño de datos más transversal es el modelo de **tenancy**. DAMA usa **esquema
compartido con columna discriminadora `TenantId`**, no una base de datos por academia.

**Justificación (IEEE 1016 exige motivar la decisión):** es la opción operativamente más simple
—una sola instancia por servicio—, con menor costo y complejidad de despliegue que "una base de
datos por tenant", y escala a muchas academias sin multiplicar la infraestructura. El **aislamiento**
no se sacrifica: se garantiza filtrando **siempre** por `TenantId` (leído del token vía
`IClaimContext`, nunca de un parámetro del cliente) en cada procedimiento almacenado, y se verifica
en pruebas y en el control de acceso (OWASP A01 / API1).

**Arista conocida y su mitigación.** Como los parámetros camelCase de los procedimientos (`tenantId`)
podrían colisionar con las columnas PascalCase (`TenantId`) —volviendo `WHERE TenantId = tenantId`
siempre verdadero—, las consultas **califican la columna con el alias de la tabla**
(`WHERE td.TenantId = tenantId`, `infrastructure/environments/auth/init.sql:133`). Es una decisión
de diseño defensiva documentada, no un detalle incidental.

## 3.3.4.3 Modelo Entidad-Relación (por servicio)

Cada base de datos con estado se modela aquí a partir de los `CREATE TABLE` de su `init.sql`. Para
cada servicio se ofrece: **(1)** una introducción que justifica el diseño del esquema, **(2)** su
representación en **DBML** —el lenguaje de [dbdiagram.io](https://dbdiagram.io); basta pegar el bloque
para obtener el diagrama entidad-relación— y **(3)** la explicación de las relaciones entre tablas.

**Convención del DBML.** Las líneas `Ref:` reflejan **claves foráneas declaradas** en el `init.sql`
(`Ref: A.x > B.y` es N:1; `A.x - B.y` es 1:1; `[delete: cascade]` transcribe el `ON DELETE CASCADE`).
Coherente con 3.3.4.1, **ninguna relación cruza servicios**: las columnas que apuntan a entidades de
otro backend (`TeacherId`, `StudentId`, `ClassId`) no son FK, sino **identificadores** validados por
gRPC o recibidos por evento. En Attendance y Payment, donde el `init.sql` **no declara FK alguna** (las
tablas se desacoplan a propósito), las relaciones intra-servicio se documentan en prosa y, cuando
ayudan a leer el diagrama, se incluyen como `Ref` anotadas explícitamente como *lógicas*
(garantizadas por la aplicación, no por el motor).

### Auth (`infrastructure/environments/auth/init.sql:6`-`53`)

**Justificación.** El esquema separa la **identidad** (`User`) de la **academia** (`Tenant`) y enlaza
ambas por una tabla puente `TenantDomain`, de modo que la pertenencia es un hecho explícito y
filtrable —es la columna `TenantId` la que materializa el aislamiento multitenant (3.3.4.2)—. El
**estado de suscripción** se extrae a `TenantAllowedServices` (1:1 con `Tenant`) en lugar de añadir
columnas a `Tenant`: así el plan se consulta y se hace *upsert* de forma independiente sin tocar los
datos básicos de la academia, y Payment puede actualizar la suscripción por gRPC sin contención sobre
`Tenant`. El borrado de usuarios es **lógico** (`IsDeleted`): la unicidad de nombre se impone sólo
entre cuentas vivas mediante la columna generada `ActiveUserName`, lo que permite reutilizar un nombre
liberado por un borrado. `RefreshToken` es una tabla satélite (1:N con `User`) que guarda únicamente el
**hash** del token para soportar rotación y revocación de sesión.

```dbml
Table User {
  Id varchar(36) [pk]
  UserName varchar(80) [not null]
  PasswordHash varchar(200) [null]
  Role varchar(50) [null, note: 'Admin / Client / Teacher / Student']
  IsDeleted tinyint(1) [not null, default: 0]
  FailedLoginAttempts int [not null, default: 0]
  LockedUntil datetime(6) [null]
  ActiveUserName varchar(80) [unique, note: 'Generada: UserName si IsDeleted=0, si no NULL']
}

Table Tenant {
  Id varchar(36) [pk]
  Name varchar(200) [null]
  Timezone varchar(64) [not null, default: 'America/La_Paz']
}

Table TenantAllowedServices {
  Id varchar(36) [pk, note: 'FK 1:1 con Tenant.Id']
  IndexCoreServicesPyramid int [not null, default: 0]
  ExpiresAt datetime(6) [not null]
}

Table TenantDomain {
  UserId varchar(36)
  TenantId varchar(36) [note: 'Aislamiento multitenant']
}

Table RefreshToken {
  Id varchar(36) [pk]
  UserId varchar(36) [not null]
  TokenHash char(64) [not null, unique]
  ExpiresAt datetime(6) [not null]
  RevokedAt datetime(6) [null]
  CreatedAt datetime(6) [not null]
}

Ref: TenantAllowedServices.Id - Tenant.Id
Ref: TenantDomain.UserId > User.Id
Ref: TenantDomain.TenantId > Tenant.Id
Ref: RefreshToken.UserId > User.Id
```

**Relaciones.** `User` y `Tenant` mantienen una relación **muchos-a-muchos** resuelta por
`TenantDomain` (un usuario puede pertenecer a una academia y la academia agrupa a muchos usuarios).
`TenantAllowedServices` extiende a `Tenant` **uno-a-uno** (comparten `Id`). `RefreshToken` pertenece a
`User` (**uno-a-muchos**). `outbox_events` (no mostrada) es una tabla técnica de mensajería sin
relación referencial, descrita en 3.3.4.5.

### CourseManagement (`infrastructure/environments/course-management/init.sql:6`-`85`)

**Justificación.** El dominio distingue dos naturalezas de clase que comparten atributos (horario,
cupo, curso, grupo) pero difieren en su **clave temporal**: la `ScheduledClass` es **recurrente** (se
repite un `DayOfWeekIndex` de la semana) y la `UniqueClass` es **puntual** (ocurre en una `Date`
concreta). Se modelan en **tablas separadas** en lugar de una tabla con columnas anulables porque cada
una tiene su propia validación de solapamiento y sus consultas por semana/fecha. `Course` agrupa
clases por materia; `ClassGroup` es la unidad sobre la que se valida el solapamiento horario (por eso
el cupo y el grupo son obligatorios en las clases). La **docencia** es **muchos-a-muchos** y se resuelve
con dos tablas puente que guardan un **snapshot desnormalizado** del nombre del profesor
(`TeacherName`): así un listado de clases no necesita un *join* cross-service contra Auth para
mostrar quién enseña. `CourseIdempotency` garantiza que una misma referencia externa no cree dos veces
la misma entidad.

```dbml
Table Course {
  Id varchar(36) [pk]
  Name varchar(200)
  TenantId varchar(36)
}

Table ClassGroup {
  Id varchar(36) [pk]
  Name varchar(200) [not null]
  TenantId varchar(36) [not null]
}

Table ScheduledClass {
  Id varchar(36) [pk]
  DayOfWeekIndex tinyint [note: 'Clase recurrente: día de la semana']
  MaxStudentLimit smallint [not null, default: 0]
  StartTime time
  EndTime time
  CourseId varchar(36)
  GroupId varchar(36) [not null]
  TenantId varchar(36) [not null]
}

Table UniqueClass {
  Id varchar(36) [pk]
  Date date [note: 'Clase puntual: fecha concreta']
  MaxStudentLimit smallint [not null, default: 0]
  StartTime time
  EndTime time
  CourseId varchar(36)
  GroupId varchar(36) [not null]
  TenantId varchar(36) [not null]
}

Table ScheduledClassTeacher {
  ScheduledClassId varchar(36) [not null]
  TeacherId varchar(36) [not null, note: 'Id de User en Auth: no es FK']
  TeacherName varchar(200) [not null, note: 'Snapshot desnormalizado']
  TenantId varchar(36) [not null]
  indexes { (ScheduledClassId, TeacherId) [pk] }
}

Table UniqueClassTeacher {
  UniqueClassId varchar(36) [not null]
  TeacherId varchar(36) [not null, note: 'Id de User en Auth: no es FK']
  TeacherName varchar(200) [not null, note: 'Snapshot desnormalizado']
  TenantId varchar(36) [not null]
  indexes { (UniqueClassId, TeacherId) [pk] }
}

Table CourseIdempotency {
  TenantId varchar(36) [not null]
  ExternalReference varchar(128) [not null]
  EntityType varchar(32) [not null]
  EntityId varchar(36) [not null]
  ProcessedAt datetime [not null]
  indexes { (TenantId, ExternalReference) [pk] }
}

Ref: ScheduledClass.CourseId > Course.Id
Ref: ScheduledClass.GroupId > ClassGroup.Id
Ref: UniqueClass.CourseId > Course.Id
Ref: UniqueClass.GroupId > ClassGroup.Id
Ref: ScheduledClassTeacher.ScheduledClassId > ScheduledClass.Id [delete: cascade]
Ref: UniqueClassTeacher.UniqueClassId > UniqueClass.Id [delete: cascade]
```

**Relaciones.** Tanto `ScheduledClass` como `UniqueClass` pertenecen a un `Course` y a un `ClassGroup`
(**muchos-a-uno** en ambos casos). La asignación de profesores es **muchos-a-muchos**: `ScheduledClassTeacher`
y `UniqueClassTeacher` enlazan cada clase con sus profesores y su clave primaria compuesta
`(ClaseId, TeacherId)` impide duplicar la misma asignación; el `ON DELETE CASCADE` borra las
asignaciones al eliminar la clase. `TeacherId` referencia a un `User` de Auth **por identificador, sin
FK** (vive en otro servicio). `CourseIdempotency` no tiene relación referencial: es un ledger de
control con clave `(TenantId, ExternalReference)`.

### Attendance (`infrastructure/environments/attendance/init.sql:6`-`175`)

**Justificación.** Attendance es deliberadamente **sin claves foráneas**: registra hechos
(asistencias) cuyos protagonistas —clases y estudiantes— viven en otros servicios y llegan por
identificador. El diseño prioriza **idempotencia** y **trazabilidad** sobre la integridad referencial.
Las asistencias se parten en dos tablas espejo de la distinción de CourseManagement (clase recurrente
vs. puntual); cada una lleva una **clave única natural** que hace imposible registrar dos veces la
misma presencia: `(TenantId, ClassId, ClassDate, StudentId)` para las recurrentes —el mismo alumno
puede asistir a la misma clase en fechas distintas— y `(TenantId, ClassId, StudentId)` para las
puntuales —que ocurren una sola vez—. Los nombres de curso y estudiante se **desnormalizan** para no
depender de otros servicios al listar. `StudentRemainClasses` mantiene el **saldo** de clases
disponibles por estudiante; las tablas `*_events`/`*_requests` son ledgers que garantizan procesar una
sola vez cada evento de pago acreditado y cada petición manual de recarga.

```dbml
Table ScheduledClassAttendance {
  TenantId varchar(36) [not null]
  ClassId varchar(36) [not null, note: 'Id de clase en CourseManagement: no es FK']
  ClassDate date [not null]
  StartTime time [not null, default: '00:00:00']
  EndTime time [not null, default: '00:00:00']
  CourseName varchar(80) [note: 'Desnormalizado']
  StudentId varchar(36) [not null, note: 'Id de User en Auth: no es FK']
  StudentName varchar(80) [note: 'Desnormalizado']
  indexes { (TenantId, ClassId, ClassDate, StudentId) [unique] }
}

Table UniqueClassAttendance {
  TenantId varchar(36) [not null]
  ClassId varchar(36) [not null]
  ClassDate date [not null, default: '1970-01-01']
  StartTime time [not null, default: '00:00:00']
  EndTime time [not null, default: '00:00:00']
  CourseName varchar(80)
  StudentId varchar(36) [not null]
  StudentName varchar(80)
  indexes { (TenantId, ClassId, StudentId) [unique] }
}

Table StudentRemainClasses {
  TenantId varchar(36) [not null]
  Id varchar(36) [not null, note: 'Id del estudiante (User en Auth)']
  NumberOfClasses int [not null, default: 0, note: 'Saldo de clases']
  StudentName varchar(80)
  indexes { (TenantId, Id) [pk] }
}

Table payment_credit_ledger {
  EventId char(36) [pk, note: 'Id del evento de pago acreditado']
  TenantId char(36) [not null]
  StudentId char(36) [not null]
  Quantity int [not null]
  ExternalReference varchar(100) [not null]
  OccurredAt datetime(6) [not null]
  CreatedAt datetime(6) [not null]
}

Table processed_events {
  EventId char(36) [pk]
  ProcessedAt datetime(6) [not null]
}

Table processed_remain_requests {
  RequestId char(36) [pk]
  ProcessedAt datetime(6) [not null]
}

// Relación LÓGICA (no es FK; garantizada por la aplicación):
// el crédito de un pago incrementa el saldo del mismo estudiante.
Ref: payment_credit_ledger.(TenantId, StudentId) > StudentRemainClasses.(TenantId, Id)
```

**Relaciones.** No hay claves foráneas. Las dos tablas de asistencia son independientes entre sí y se
relacionan con el resto del sistema **sólo por identificador** (`ClassId` → clase de CourseManagement,
`StudentId` → usuario de Auth). La única relación intra-servicio relevante es **lógica**: cuando llega
un evento de pago, `payment_credit_ledger` deja constancia y se incrementa el saldo del estudiante
correspondiente en `StudentRemainClasses` (join por `(TenantId, StudentId)` ↔ `(TenantId, Id)`).
`processed_events` y `processed_remain_requests` son ledgers de idempotencia puros, sin relación con
las demás tablas.

### Payment (`infrastructure/environments/payment/init.sql:6`-`480`)

**Justificación.** Payment es el servicio con el modelo más rico porque combina un **catálogo**, una
**máquina de estados de pago** y la **mensajería transaccional**. Los pagos no se modelan como una
tabla con una columna `Estado`, sino como **ledgers separados por estado** (`Pending` / `Success` /
`Failed`): un pago nace en la tabla *Pending* y, al resolverse, se materializa en *Success* o *Failed*
**conservando su `Id`**; esto da tablas inmutables por estado, consultas e índices específicos
(p. ej. cobros del mes sobre `SuccessQrPayment`) y un historial a prueba de mutaciones. El patrón se
aplica **dos veces**: a los pagos QR del **estudiante** (compra de clases contra un `DebtTemplate`) y a
los pagos de **suscripción** de la **academia** a DAMA (contra un `SubscriptionPlan`). `DebtTemplate`
cataloga las deudas que una academia ofrece; `TenantPaymentCredentials` guarda la clave Todotix
**cifrada** por academia; `SubscriptionPlan` es el catálogo (editable por Admin) de precio y duración
por nivel de pirámide. El resto son **tablas técnicas** de outbox/inbox e idempotencia que sacan las
llamadas a Todotix y la publicación de eventos fuera de la ruta de la petición.

```dbml
Table DebtTemplate {
  Id varchar(36) [pk]
  TenantId varchar(36) [not null]
  Description varchar(256) [not null]
  ClassQuantity int [not null]
  Cost int [not null]
  Currency char(3) [not null, default: 'BOB']
}

Table TenantPaymentCredentials {
  TenantId varchar(36) [pk]
  TodotixAppKey varchar(512) [not null, note: 'Clave Todotix cifrada']
}

Table PendingQrPayment {
  Id varchar(36) [pk]
  TenantId varchar(36) [not null]
  StudentId varchar(36) [not null, note: 'Id de User en Auth: no es FK']
  TemplateId varchar(36) [not null]
  ClassQuantity int [not null]
  Cost int [not null]
  Currency char(3) [not null, default: 'BOB']
  QrImageUrl varchar(512) [null]
  CreatedAt datetime [not null]
  ExpiresAt datetime(6) [not null]
}

Table SuccessQrPayment {
  Id varchar(36) [pk, note: 'Mismo Id que el PendingQrPayment de origen']
  TenantId varchar(36) [not null]
  StudentId varchar(36) [not null]
  ClassQuantity int [not null]
  Cost int [not null]
  Currency char(3) [not null, default: 'BOB']
  PaidAt datetime [not null]
}

Table FailedQrPayment {
  Id varchar(36) [pk, note: 'Mismo Id que el PendingQrPayment de origen']
  TenantId varchar(36) [not null]
  StudentId varchar(36) [not null]
  ClassQuantity int [not null]
  Cost int [not null]
  Currency char(3) [not null, default: 'BOB']
  FailedAt datetime [not null]
  FailureReason "ENUM('Expired','CallbackError','Manual')" [not null, default: 'CallbackError']
}

Table SubscriptionPlan {
  Level int [pk, note: 'Nivel de la pirámide de servicios']
  Price int [not null]
  Currency char(3) [not null, default: 'BOB']
  DurationAmount int [not null]
  DurationUnit varchar(8) [not null, default: 'Month']
  UpdatedAt datetime(6) [not null]
}

Table PendingSubscriptionPayment {
  Id char(36) [pk]
  TenantId char(36) [not null]
  Level int [not null]
  Cost int [not null]
  Currency char(3) [not null, default: 'BOB']
  QrImageUrl varchar(512) [null]
  CreatedAt datetime(6) [not null]
  ExpiresAt datetime(6) [not null]
}

Table SuccessSubscriptionPayment {
  Id char(36) [pk, note: 'Mismo Id que el PendingSubscriptionPayment de origen']
  TenantId char(36) [not null]
  Level int [not null]
  Cost int [not null]
  Currency char(3) [not null, default: 'BOB']
  PaidAt datetime(6) [not null]
}

Table FailedSubscriptionPayment {
  Id char(36) [pk, note: 'Mismo Id que el PendingSubscriptionPayment de origen']
  TenantId char(36) [not null]
  Level int [not null]
  Cost int [not null]
  Currency char(3) [not null, default: 'BOB']
  FailedAt datetime(6) [not null]
}

Table todotix_outbox {
  Id char(36) [pk]
  PendingId char(36) [not null]
  TenantId char(36) [not null]
  DebtKind varchar(24) [not null, default: 'ClassPurchase']
  PayloadJson json [not null]
  OccurredAt datetime(6) [not null]
  ProcessedAt datetime(6) [null]
  LeasedUntil datetime(6) [null]
  Attempts int [not null, default: 0]
  LastError varchar(500) [null]
  Status varchar(16) [not null, default: 'Pending']
}

Table QrPaymentIdempotency {
  TenantId varchar(36) [not null]
  ExternalReference varchar(128) [not null]
  EntityId varchar(36) [not null]
  ProcessedAt datetime [not null]
  indexes { (TenantId, ExternalReference) [pk] }
}

// Relaciones LÓGICAS (no son FK; garantizadas por la aplicación):
Ref: PendingQrPayment.TemplateId > DebtTemplate.Id
Ref: todotix_outbox.PendingId > PendingQrPayment.Id
Ref: PendingSubscriptionPayment.Level > SubscriptionPlan.Level
Ref: SuccessSubscriptionPayment.Level > SubscriptionPlan.Level
Ref: FailedSubscriptionPayment.Level > SubscriptionPlan.Level
```

**Relaciones.** El `init.sql` **no declara FK**: todas las relaciones son lógicas. Un pago QR de
estudiante se origina en `PendingQrPayment` apuntando (por `TemplateId`) al `DebtTemplate` que define
cantidad y costo, y al resolverse pasa a `SuccessQrPayment` o `FailedQrPayment` con **el mismo `Id`**
—los tres ledgers comparten clave primaria a lo largo del ciclo de vida del pago—. El mismo trío
`Pending`/`Success`/`Failed` se repite para los pagos de **suscripción** de la academia, que apuntan
(por `Level`) al `SubscriptionPlan`. `todotix_outbox` relaciona cada deuda a registrar con su
`PendingQrPayment` de origen (`PendingId`). `StudentId`/`TenantId` referencian entidades de Auth por
identificador. Las tablas técnicas restantes (`expiration_outbox`, `outbox_events`,
`payment_callback_inbox`, `processed_events`, `QrPaymentIdempotency`) operan a nivel de infraestructura
de mensajería e idempotencia; el patrón de los **cuatro ledgers** de Payment se detalla en el patrón #9
de `apps/CLAUDE.md`.

| `Id` | VARCHAR(36) | No | PK | Identificador del token. |
| `UserId` | VARCHAR(36) | No | FK → `User(Id)` | Usuario dueño del token. |
| `TokenHash` | CHAR(64) | No | UNIQUE | Hash del token (no se guarda el valor en claro). |
| `ExpiresAt` | DATETIME(6) | No | — | Expiración del token. |
| `RevokedAt` | DATETIME(6) | Sí | — | Instante de revocación (al cerrar sesión), si aplica. |
| `CreatedAt` | DATETIME(6) | No | — | Instante de emisión. |

## 3.3.4.5 Patrón de columna `TenantId` en el resto de servicios

En CourseManagement, Attendance y Payment, **toda tabla de negocio** lleva `TenantId VARCHAR(36)`
(p. ej. `Course.TenantId`, `ScheduledClass.TenantId`, `DebtTemplate.TenantId`,
`StudentRemainClasses.TenantId`), normalmente indexado, con idéntica semántica: *academia propietaria
del registro, base del aislamiento*. La excepción son las tablas técnicas de mensajería
(`outbox_events`, `processed_events`) en `snake_case`, que operan a nivel de infraestructura.
