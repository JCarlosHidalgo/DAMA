## 3.5.2.4 Política de cobertura

La cobertura se mide sobre la **lógica de negocio**, no sobre la infraestructura: el `.runsettings`
de las cuatro suites backend excluye `Controllers/`, `DB/`, `Entities/`, `Events/`, `Filters/`,
`Messaging/`, `Modules/`, `Options/`, `Workers/` y `Program.cs` (5.1.2), de modo que el porcentaje
refleja la prueba de la lógica y no se infla con código de cableado. En el frontend, la barrera de
cobertura crítica **exige 100 %** en las rutas de lógica, validadores y *stores* (con una sola
relajación de ramas para los *pipes* compartidos). El porcentaje exacto por servicio se obtiene con
el recolector (`dotnet test -s ./.runsettings`), cuya configuración es la citada; aquí se reporta el
**resultado de las suites** (1 251/1 251) y la **vigencia de las barreras**, no un número de
cobertura estimado.

## 3.5.2.5 Comandos de demostración

```bash
# Suites backend (cifras de 5.2.1) — desde la raíz
for s in Auth Attendance CourseManagement Payment; do ( cd apps/$s/Test && dotnet test -c Release --nologo ); done

# Suite del frontend
cd apps/Frontend && bun run test:ci

# Conteo de archivos de prueba por suite
for s in Auth Attendance CourseManagement Payment; do \
  printf "%-18s %s\n" "$s" "$(find apps/$s/Test -name '*Tests.cs' -not -path '*/obj/*' | wc -l)"; done

# Cobertura de lógica con la configuración del repo (porcentaje por servicio)
cd apps/Auth/Test && dotnet test -s ./.runsettings

# Cobertura crítica del frontend (barrera al 100 %)
cd apps/Frontend && bun run test:coverage:gate
```
