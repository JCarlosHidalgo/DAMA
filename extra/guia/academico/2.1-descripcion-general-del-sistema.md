## 3.2.1.1 Propósito

DAMA es una **plataforma SaaS multitenant para la gestión integral de academias de baile**. Su
propósito es ofrecer a cada academia (un *tenant*) un único sistema en línea para administrar su
oferta académica —cursos, clases y grupos—, registrar la **asistencia** de los estudiantes y
gestionar los **cobros** a través de una pasarela de pagos externa, todo bajo un modelo de
**suscripción por niveles** que habilita o restringe funcionalidades según el plan contratado.

Siguiendo la recomendación de ISO/IEC/IEEE 29148:2018 de enunciar el propósito en términos del
beneficio que aporta y no de la tecnología que lo implementa, el sistema resuelve tres problemas
operativos recurrentes de una academia: (1) centralizar la programación de clases y la información
de profesores y estudiantes; (2) llevar un control de asistencia fiable y a prueba de
suplantación; y (3) integrar el cobro de mensualidades y deudas con un proveedor de pagos sin
exponer las credenciales financieras de cada academia.

## 3.2.1.2 Alcance y delimitación

El alcance del producto se delimita por los dominios efectivamente implementados en el
repositorio, cada uno encarnado en un microservicio con su propia base de datos
(`README.md:5`-`16`):

| Dominio (servicio) | Dentro del alcance | Fuera del alcance |
|---|---|---|
| **Auth** | Registro y autenticación de usuarios, gestión de academias (tenants), niveles de suscripción, multitenancy. | Federación de identidad externa (OAuth de terceros, SSO corporativo). |
| **CourseManagement** | Cursos, clases **programadas** (recurrentes) y **únicas**, grupos de clase, horarios por rol. | Gestión de aulas físicas o recursos materiales. |
| **Attendance** | Toma de asistencia (incluida la vía QR), clases restantes (*remain*) por estudiante. | Geolocalización o biometría. |
| **Payment** | Registro y cobro de deudas vía **Todotix**, pagos QR, pago de suscripciones, plantillas de deuda, analítica de ingresos. | Procesamiento directo de tarjetas o custodia de fondos (lo realiza Todotix). |
| **Credentials** | Reflexión de *claims* del usuario autenticado (servicio sin estado, sin base de datos). | Almacenamiento de credenciales propias. |

Queda **explícitamente fuera del alcance** todo lo que el código no implementa: una aplicación
móvil nativa (DAMA es una SPA web —ver `extra/OWASP/README.md:21`), funcionalidades de IA
generativa (`extra/OWASP/README.md:22`), y la operación financiera de bajo nivel, delegada a la
pasarela externa Todotix.

## 3.2.1.3 Perspectiva del producto

DAMA no es un sistema monolítico aislado, sino un **conjunto de microservicios desacoplados** que
colaboran detrás de una única puerta de enlace. La perspectiva del producto —subsección 2 del SRS
en 29148— describe el contexto en el que opera y los sistemas con los que interactúa
(`README.md:7`-`19`):

- **Frontend (SPA Angular 21):** el único cliente humano. Se sirve como contenido estático desde
  Apache httpd y consume la API exclusivamente a través del gateway.
- **api-gateway (nginx):** el **único punto de entrada** público. Enruta cada petición REST al
  microservicio correspondiente, aplica *rate-limiting*, compresión y límites de tamaño de
  petición (ver 3.2.4), y termina siendo el lugar donde viven los controles transversales de borde.
- **Cinco microservicios .NET 9** (Auth, CourseManagement, Attendance, Payment, Credentials), cada
  uno dueño de su propia base de datos MySQL —salvo Credentials, que no tiene estado.
- **Comunicación entre servicios**, de dos naturalezas complementarias:
  - **Síncrona (gRPC sobre TLS)** cuando el llamante debe bloquearse a esperar una respuesta:
    `Attendance → CourseManagement` (verificación de existencia de clase/curso) y
    `Payment → Auth` (aplicación del nivel de suscripción tras un pago).
  - **Asíncrona (Outbox → RabbitMQ)** para la propagación de eventos de dominio sin acoplar las
    transacciones: `student.registered`, `course.deleted`, `class.deleted`, `payment.captured`.
- **Sistema externo Todotix:** la pasarela de pagos. Payment registra deudas y consulta su estado
  vía HTTPS, usando credenciales **cifradas por tenant**.

Esta arquitectura por contenedores, con un gateway como frontera y mensajería asíncrona como
columna vertebral de la integración, es la que se detalla en la sección de diseño (3.3.1) y se
representa en la **vista isométrica de arquitectura** generada con FossFlow.

## 3.2.1.5 Comandos de demostración

Comandos de solo lectura que permiten **verificar sobre el repositorio** la composición del sistema
descrita. Se ejecutan desde la raíz del repositorio.

**Confirmar los cinco backends más el frontend:**

```bash
ls apps/                       # Auth Attendance CourseManagement Payment Credentials Frontend
```

**Leer la descripción y el diagrama de arquitectura textual del repositorio:**

```bash
sed -n '1,25p' README.md       # propósito, arquitectura y flujos de integración
```

**Verificar que cada backend de negocio posee su propia base de datos (init.sql por servicio):**

```bash
ls infrastructure/environments/*/init.sql   # auth, attendance, course-management, payment (Credentials no tiene)
```

**Abrir el diagrama de contexto (vista isométrica):**

```bash
cd extra/fossflow && docker compose up --build   # http://localhost:8088 → cargar "dama-architecture"
```
